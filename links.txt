You can’t get a pixel-perfect, frame-by-frame copy of a live iframe because:  

• Remotion captures exactly one “snapshot” per output frame. Any CSS/JS animation inside the iframe still runs at real-time speed, so if it updates faster than your FPS, intermediate states get skipped.   

• The iframe’s own clock isn’t hooked into Remotion’s frame clock—Remotion only “freezes” and captures whatever the browser shows at each sampled instant.   

• Loading depends on network/server latency. Remotion wraps the iframe in a delayRender() with a timeout/retry policy, but slow connections can still lead to blank or loading frames. 
  
• Running on Lambda doesn’t change this: it’s still one snapshot per frame against a real-time page load. To guarantee every step you’d need to drive the animation via Remotion (useCurrentFrame/interpolate) or pre-record it as a video.





Unfortunately, you can’t perfectly sync a live iframe’s animation to Remotion frames if you don’t know its total duration up front—Remotion only samples one snapshot per output frame, so any sub-frame updates slip through. To match every step you’ll need to either: 

1) Drive the animation via Remotion’s frame clock (e.g. useCurrentFrame() + interpolate()) so you control its timing exactly.  
2) Pre-record the iframe as a video (e.g. with Puppeteer/Playwright + FFmpeg), import it with <Video> or <OffthreadVideo>, and then Remotion can step through every frame deterministically.
